# 向量检索原理解释

## 🔄 工作流程

### 索引构建阶段（写入）

```
Memory 文件 (.md)
    ↓
分块（200 tokens/块）
    ↓
Qwen3 Embedding 模型（Ollama）
    ↓
生成向量（1024 维）
    ↓
SQLite 数据库（存储向量 + 元数据）
```

### 搜索阶段（读取）

```
用户查询文本
    ↓
Qwen3 Embedding 模型（Ollama）
    ↓
生成查询向量（1024 维）
    ↓
SQLite 数据库（向量相似度检索）
    ↓
返回最相关的文本片段
```

## 📊 SQLite 数据库与向量模型的关系

### SQLite 数据库存储的内容

```sql
-- 简化的表结构
CREATE TABLE embeddings (
  id INTEGER PRIMARY KEY,
  chunk_text TEXT,           -- 原始文本片段
  embedding BLOB,            -- 向量（1024 维浮点数）
  file_path TEXT,            -- 来源文件
  line_start INTEGER,        -- 起始行号
  line_end INTEGER,          -- 结束行号
  timestamp INTEGER          -- 时间戳
);
```

**实际表结构：**
- `meta` - 元数据（模型信息、版本）
- `files` - 文件列表（MEMORY.md, memory/*.md）
- `chunks` - 文本块 + 向量
- `embedding_cache` - 向量缓存
- `chunks_fts` - 全文搜索索引（文本检索）

### 向量模型的作用

**Qwen3 Embedding 模型做两件事：**

1. **索引时：** 把文本转换成向量
   ```
   输入："OpenClaw 本地记忆系统配置"
   输出：[0.123, -0.456, 0.789, ..., 0.321]  // 1024 个数字
   ```

2. **搜索时：** 把查询转换成向量
   ```
   输入："如何配置本地记忆"
   输出：[0.145, -0.432, 0.801, ..., 0.298]  // 1024 个数字
   ```

然后在 SQLite 中计算向量相似度（余弦相似度）找到最相关的文本。

## 🎯 关键概念

### 1. 向量模型（Qwen3）= 翻译器

- 把文本翻译成数字（向量）
- 语义相似的文本 → 相似的向量
- 例如："配置" 和 "设置" 语义相似，向量也相似

### 2. SQLite 数据库 = 图书馆

- 存储所有向量和原始文本
- 快速查找相似的向量
- 支持混合检索（向量 + 文本）

### 3. 为什么需要向量？

- 文本无法直接比较相似度
- 向量可以用数学方法计算相似度
- 支持语义搜索（理解意思，而不只是关键词匹配）

### 4. 为什么用 SQLite？

- ✅ 轻量级（不需要额外数据库服务）
- ✅ 快速（本地文件访问）
- ✅ 可靠（ACID 事务）
- ✅ 跨平台（macOS/Linux/Windows）

## 📐 向量相似度计算

### 余弦相似度公式

```
similarity = (A · B) / (||A|| × ||B||)

其中：
- A 是查询向量
- B 是存储的向量
- · 表示点积
- ||A|| 表示向量 A 的模（长度）
```

### 相似度范围

- `1.0` - 完全相同
- `0.7-0.9` - 非常相似（通常是相关内容）
- `0.4-0.7` - 中等相似（可能相关）
- `0.0-0.4` - 不太相似
- `0.0` - 完全不相关

### 实际例子

```
查询："如何配置 OpenClaw"

结果：
1. "OpenClaw 配置指南" (0.85) ✅ 高度相关
2. "本地记忆系统设置" (0.72) ✅ 相关
3. "安装 Ollama 步骤" (0.58) ⚠️ 中等相关
4. "天气预报 API" (0.15) ❌ 不相关
```

## 🔄 混合检索策略

OpenClaw 使用混合检索（Hybrid Search）结合两种方法：

### 1. 向量检索（60%）

- 基于语义相似度
- 理解意思，不只是关键词
- 例如："配置" 和 "设置" 会被认为相似

### 2. 文本检索（40%）

- 基于关键词匹配（BM25 算法）
- 精确匹配重要术语
- 例如："OpenClaw" 必须出现在结果中

### 为什么混合？

- 向量检索：理解语义，但可能漏掉重要关键词
- 文本检索：精确匹配，但不理解同义词
- 混合检索：两者优势结合，效果最好

## 📊 性能优化

### 向量维度的影响

- **1024 维**（Qwen3）：平衡精度和速度
- 更高维度（2048）：更精确，但更慢
- 更低维度（512）：更快，但精度降低

### 数据库大小估算

```
每个文本块：
- 文本：~200 tokens ≈ 800 字节
- 向量：1024 维 × 4 字节 = 4,096 字节
- 元数据：~200 字节
- 总计：~5,000 字节/块

100 个文件 × 50 块/文件 = 5,000 块
5,000 块 × 5,000 字节 = 25 MB
```

### 搜索性能

- **索引构建**：~30 秒（100 个文件）
- **单次搜索**：~200ms
- **瓶颈**：向量模型推理（Ollama）

## 🔧 调优建议

### 提高精度

1. 增加 `vectorWeight`（如 0.7）
2. 降低 `minScore`（如 0.2）
3. 增加 `maxResults`（如 15）

### 提高速度

1. 减少 `maxResults`（如 5）
2. 使用更小的模型（nomic-embed-text）
3. 限制 Ollama 并发数

### 平衡精度和速度

1. 使用默认配置（vectorWeight: 0.6, textWeight: 0.4）
2. 启用 MMR 去重（lambda: 0.7）
3. 启用时间衰减（halfLifeDays: 60）
